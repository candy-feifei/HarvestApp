# HarvestApp · 前端单元测试说明

本文说明 `frontend/` 中基于 **Vite 6** 的**单测技术选型、目录约定、命令与 Mock 策略**，与 `src/lib/api/http.ts` 及 `features/*/api.ts` 的对接方式一致。

---

## 1. 技术栈

| 组件 | 说明 |
|------|------|
| **Vitest** | 与 Vite 同构配置、跑得快，运行器为 `package.json` 的 `test` / `test:run` |
| **jsdom** | 在 Node 中模拟 DOM，供组件测试使用 |
| **@testing-library/react** | 以用户视角查询节点（`getByRole` 等），避免测实现细节 |
| **@testing-library/jest-dom** | 扩展断言（如 `toBeInTheDocument()`），在 `vitest.setup.ts` 中接入 |
| **@testing-library/user-event**（可选） | 模拟键盘、点击等，适合表单与交互用例 |

Vitest 的测试选项写在 **`vite.config.ts`** 的 `test` 段（使用 `import { defineConfig } from 'vitest/config'`，以支持 `test` 字段类型）。

---

## 2. npm 脚本

| 命令 | 作用 |
|------|------|
| `npm test` | 启动 Vitest ** watch 模式**（改代码会重跑） |
| `npm run test:run` | **单次**跑完所有单测（适合 CI 或提交前自测） |
| `npm run test:tasks` | 只跑 `src/features/tasks` 下测试（示例范围限定） |

---

## 3. 目录与文件约定

| 类型 | 约定 | 说明 |
|------|------|------|
| 工具/纯函数 | 与源文件同目录，如 `api.test.ts` 与 `api.ts` 并列 | 文件名 `*.test.ts` |
| React 组件/页面 | `*.test.tsx` 与组件同目录，如 `pages/tasks-page.test.tsx` | 由 `vitest` 的 `include` 收集 |

`vite.config.ts` 中当前配置为：

- `include`：`src/**/*.test.ts`、`src/**/*.test.tsx`
- `setupFiles`：仓库根下 **`vitest.setup.ts`**（建议与 `vite.config.ts` 同级，便于复用路径别名）
- `environment`：`jsdom`
- 路径别名 **`@` → `src`**：在 Vitest 中与开发构建一致，可直接 `import` `@/...`

---

## 4. Mock 策略

### 4.1 只测 `features/*/api.ts` 网络封装（推荐首测）

- **目标**：`listTasks`、`createTask` 等是否拼出正确路径、是否以 `GET`/`POST` 等调用 `apiRequest`。
- **做法**：`vi.mock('@/lib/api/http', () => ({ apiRequest: vi.fn(), ... }))`，在测试中 `mockResolvedValue` / 断言**调用参数**。
- **不**在单测里发真实 `fetch`，避免环境变量与 CORS 干扰。

### 4.2 测带 TanStack Query 的页面

- 使用 **`QueryClientProvider`**，并在 `defaultOptions` 中关闭或收紧 **`queries.retry`**，避免用例里长时间重试。
- 对 `listTasks` 等可 **`vi.mock('@/features/xxx/api', importOriginal)`** 保留其余导出、只 stub 列表函数。

### 4.3 全链路/契约（可选加强）

- 需模拟接口响应形状时，可引入 **MSW（Mock Service Worker）** 在 Node 中拦截；当前仓库**未**强制，可按需再加依赖与 `setupFiles`。

### 4.4 ESLint

- `**/*.{test,spec}.ts(x)` 与 `vitest.setup.ts` 在 `eslint.config.js` 中启用了与 Vitest 相关的 **globals**，并关闭 `react-refresh/only-export-components` 对测试文件的限制。

---

## 5. CI 建议

- 流水线中执行 **`npm run test:run`**（非交互、非 watch）。
- 与 `npm run build` 一并跑，可尽早发现仅测试环境暴露的问题（如缺少 polyfill、环境变量未注入）。

---

## 7. 与《前端搭建说明》的关系

单测不替代 **联调**（`VITE_API_BASE_URL`、`vite` 代理到本地 Nest）。开发时仍以后端实跑、前端 `npm run dev` 为主；单测保障「在假数据/假网络下，组件与 `api` 层行为可重复、可回归」。
