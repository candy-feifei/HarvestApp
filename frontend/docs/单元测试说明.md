# HarvestApp · 前端单元测试说明

本文说明 `frontend/` 中基于 **Vite 6** 的**单测技术选型、目录约定、命令与 Mock 策略**，与 `src/lib/api/http.ts` 及 `features/*/api.ts` 的对接方式一致。

---

## 1. 技术栈

| 组件 | 说明 |
|------|------|
| **Vitest** | 与 Vite 同构配置、跑得快，运行器为 `package.json` 的 `test` / `test:run` |
| **@vitest/coverage-v8** | 与 Vitest 主版本对齐的 V8 覆盖率提供器，用于 `test:coverage` 等 |
| **jsdom** | 在 Node 中模拟 DOM，供组件测试使用 |
| **@testing-library/react** | 以用户视角查询节点（`getByRole` 等），避免测实现细节 |
| **@testing-library/jest-dom** | 扩展断言（如 `toBeInTheDocument()`），在 `vitest.setup.ts` 中接入 |
| **@testing-library/user-event**（可选） | 模拟键盘、点击等，适合表单与交互用例 |

Vitest 的测试选项写在 **`vite.config.ts`** 的 `test` 段（使用 `import { defineConfig } from 'vitest/config'`，以支持 `test` 字段类型）。覆盖率在 **`test.coverage`** 中配置：`provider: 'v8'`、`reporter`（`text` / `html` / `json`）、`reportsDirectory` 指向 `frontend/coverage`（见**第 6 节**）。

---

## 2. npm 脚本

| 命令 | 作用 |
|------|------|
| `npm test` | 启动 Vitest ** watch 模式**（改代码会重跑） |
| `npm run test:run` | **单次**跑完所有单测（适合 CI 或提交前自测） |
| `npm run test:tasks` | 只跑 `src/features/tasks` 下测试（不收集覆盖率） |
| `npm run test:coverage` | 全量单测 + **代码覆盖率**（V8，报告见**第 6 节**） |
| `npm run test:coverage:tasks` | 仅 **tasks 模块** 的用例 + 覆盖率；报告**只统计** `src/features/tasks/**` 下源码（与当前仅 tasks 有单测的场景一致，避免全项目大量 0%） |

当前**已编写单测的 feature** 以 **tasks** 为主；其他模块补充测试后，可仿照 `test:coverage:tasks` 增加 `vitest run --coverage --coverage.include=… <路径>` 的脚本。

---

## 3. 目录与文件约定

| 类型 | 约定 | 说明 |
|------|------|------|
| 工具/纯函数 | 与源文件同目录，如 `api.test.ts` 与 `api.ts` 并列 | 文件名 `*.test.ts` |
| React 组件/页面 | `*.test.tsx` 与组件同目录，如 `pages/tasks-page.test.tsx` | 由 `vitest` 的 `include` 收集 |

`vite.config.ts` 中当前配置为：

- `include`：`src/**/*.test.ts`、`src/**/*.test.tsx`
- `setupFiles`：仓库根下 **`vitest.setup.ts`**（建议与 `vite.config.ts` 同级，便于复用路径别名）
- `environment`：`jsdom`
- 路径别名 **`@` → `src`**：在 Vitest 中与开发构建一致，可直接 `import` `@/...`

## 4. Mock 策略

### 4.1 只测 `features/*/api.ts` 网络封装（推荐首测）

- **目标**：`listTasks`、`createTask` 等是否拼出正确路径、是否以 `GET`/`POST` 等调用 `apiRequest`。
- **做法**：`vi.mock('@/lib/api/http', () => ({ apiRequest: vi.fn(), ... }))`，在测试中 `mockResolvedValue` / 断言**调用参数**。
- **不**在单测里发真实 `fetch`，避免环境变量与 CORS 干扰。

### 4.2 测带 TanStack Query 的页面

- 使用 **`QueryClientProvider`**，并在 `defaultOptions` 中关闭或收紧 **`queries.retry`**，避免用例里长时间重试。
- 对 `listTasks` 等可 **`vi.mock('@/features/xxx/api', importOriginal)`** 保留其余导出、只 stub 列表函数。

### 4.3 全链路/契约（可选加强）

- 需模拟接口响应形状时，可引入 **MSW（Mock Service Worker）** 在 Node 中拦截；当前仓库**未**强制，可按需再加依赖与 `setupFiles`。

### 4.4 ESLint

- `**/*.{test,spec}.ts(x)` 与 `vitest.setup.ts` 在 `eslint.config.js` 中启用了与 Vitest 相关的 **globals**，并关闭 `react-refresh/only-export-components` 对测试文件的限制。

---

## 5. CI 建议

- 流水线中执行 **`npm run test:run`**（非交互、非 watch）；若需门禁覆盖率，再增加 **`npm run test:coverage`** 或分模块的 `test:coverage:*`。
- 与 `npm run build` 一并跑，可尽早发现仅测试环境暴露的问题（如缺少 polyfill、环境变量未注入）。

## 6. 覆盖率与可视化报告

### 6.1 依赖与配置

- **依赖**：`devDependencies` 中的 **`@vitest/coverage-v8`**（版本与 `vitest` 主版本对齐，如 `^3.2.4`）。
- **配置**：`vite.config.ts` → `test.coverage`：`provider: 'v8'`，`reporter` 含 `text`（终端表）、`html`（浏览）、`json`（工具链）；`reportsDirectory` 为项目下 **`coverage/`**（已写入 **`frontend/.gitignore`**，勿提交报告产物）。

### 6.2 全量覆盖率

| 说明 | 内容 |
|------|------|
| 命令 | `npm run test:coverage`（即 `vitest run --coverage`） |
| HTML 报告 | `frontend/coverage/index.html`，用浏览器打开 |
| 终端 | 自动输出摘要表 |

在仅有 tasks 有单测时，全量模式会对**未覆盖的其它 `src` 文件**也计入报表（多为 0%）；需要「只看某模块」时见下一小节。

### 6.3 只跑 / 只统计某模块（示例：tasks）

- **只跑用例、不要覆盖率**：`npm run test:tasks` 或对 Vitest 传入路径，如 `vitest run src/features/xxx`。
- **只要 tasks 的覆盖率**（**统计范围**限定在 `src/features/tasks/**`）：`npm run test:coverage:tasks`  
  实现上等价于：`vitest run --coverage --coverage.include=src/features/tasks/** src/features/tasks`  
  这样 HTML/JSON 报告**仅包含**该目录下源码的覆盖情况，与「当前只有 tasks 有单测」的迭代方式一致。

其他目录写法示例（自行加到 `package.json` 或临时执行）：

```bash
npx vitest run --coverage --coverage.include=src/features/其它目录/** src/features/其它目录
```

### 6.4 交互式用例与依赖图：Vitest UI

| 包名 | 说明 |
|------|------|
| **`@vitest/ui`** | 安装后执行如 **`npx vitest --ui`**，在浏览器中查看**当前运行/历史**的用例树、时间线、部分模块依赖，偏开发时调试，不是单文件「静态报告」但可视化程度高 |

### 6.5 用例结果的 HTML 报表（可选）

Vitest 支持多种 **reporter**（如终端、`json`、`junit`）。要 **HTML 形式的用例通过/失败汇总**，可在 `vite.config` → `test.reporters` 中查阅官方支持的 **`html` reporter**（及输出目录，随版本以 [Reporters 文档](https://vitest.dev/config/#reporter) 为准），或接 CI 里的 Sonar 等。当前仓库**未**强制这些选项，按团队需求添加即可。

更细的 coverage CLI（如 `coverage.all`、`thresholds`）见 [Vitest — Coverage](https://vitest.dev/guide/coverage.html)。

## 8. 与《前端搭建说明》的关系

单测不替代 **联调**（`VITE_API_BASE_URL`、`vite` 代理到本地 Nest）。开发时仍以后端实跑、前端 `npm run dev` 为主；单测保障「在假数据/假网络下，组件与 `api` 层行为可重复、可回归」。
