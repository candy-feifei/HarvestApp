# HarvestApp 前端搭建说明

本文档说明本仓库 `frontend/` 从零到当前形态的搭建过程、关键决策与协作时注意点，便于复盘与答辩材料引用。

---

## 1. 技术选型

| 层级 | 选型 | 说明 |
|------|------|------|
| 构建 | Vite 6.x + `@vitejs/plugin-react` | 未使用 Vite 8：部分环境下其默认依赖 `rolldown` 的原生绑定可能缺失，导致 `vite build` 失败；Vite 6 更稳。 |
| 框架 | React 19 + TypeScript | 与 NestJS 同用 TS 生态，便于接口契约与 AI 辅助编码。 |
| 样式 | Tailwind CSS v4 + `@tailwindcss/vite` | 在 `vite.config.ts` 中注册插件；全局样式入口为 `src/index.css`。 |
| 组件库 | shadcn/ui（Radix Nova 预设） | 通过 `npx shadcn@latest init --template vite -y -b radix -p nova` 初始化；组件源码在仓库内，便于定制。 |
| 路由 | React Router 7 | `createBrowserRouter` + 按模块 `lazy()` 懒加载。 |
| 服务端状态 | TanStack Query v5 | 仅作「请求缓存 / 重试 / loading」等，**未引入** Redux / Zustand 等全局业务 Store。 |
| 后端对接 | `fetch` 封装 `apiRequest`（`src/lib/api/http.ts`） | 与 Nest 返回的 JSON 错误体做简单归一；鉴权头后续可集中注入。 |

## 2. 初始化步骤（按时间顺序）

### 2.1 创建 Vite 工程

在项目根目录执行（示例）：

```bash
npm create vite@latest frontend -- --template react-ts
cd frontend
npm install
```

### 2.2 接入 Tailwind CSS v4

```bash
npm install tailwindcss @tailwindcss/vite
```

在 `vite.config.ts` 中：

- 增加 `import tailwindcss from '@tailwindcss/vite'`，并加入 `plugins: [react(), tailwindcss()]`。
- 配置路径别名 `@` → `src`（使用 `import.meta.url` 解析 `__dirname`，避免 ESM 下未定义问题）。

在 `src/index.css` 中保留（或合并）Tailwind 入口；shadcn 初始化后会写入主题与字体相关 `@import`。

### 2.3 配置 TypeScript 路径别名

- 在 **`tsconfig.json`** 根配置中增加 `compilerOptions.baseUrl` 与 `paths["@/*"]`：**shadcn CLI 会校验根 tsconfig 是否含别名**。
- 在 **`tsconfig.app.json`** 中同步 `paths`；若使用 TypeScript 6 且出现 `baseUrl` 弃用提示，可临时增加 `"ignoreDeprecations": "6.0"`（以官方迁移指南为准逐步调整）。

### 2.4 初始化 shadcn/ui

```bash
cd frontend
npx shadcn@latest init --template vite -y -b radix -p nova
```

说明：

- 需已安装 Tailwind v4 且通过 shadcn 预检。
- 若仅执行 `init` 仍出现交互选项，可显式传入 `-b radix -p nova` 等非交互参数。

初始化完成后会生成 `components.json`、`src/lib/utils.ts`，并通常安装首个组件（如 `button`）。

### 2.5 安装路由与数据层

```bash
npm install react-router-dom @tanstack/react-query
```

### 2.6 环境变量类型

新增 `src/vite-env.d.ts`，为 `import.meta.env.VITE_API_BASE_URL` 等提供类型声明。

## 3. 目录与架构演进（第二次优化）

在「壳层能跑」的基础上，将代码调整为 **按领域（feature）竖切**，便于与 Harvest 模块及 Nest 模块一一对应、分工开发：

1. **`src/features/<领域>/`**  
   每个领域包含 `pages/`（路由页面）、`api.ts`（与 Nest 资源路径对齐的占位或真实请求）。
2. **`src/app/`**  
   `root.tsx`（挂载 `AppProviders` + `RouterProvider`）、`providers.tsx`（`QueryClientProvider`）、`router.tsx`（路由表与 `lazy`）。
3. **`src/lib/api/`**  
   `public-env.ts`（读取 `VITE_API_BASE_URL`）、`http.ts`（`apiRequest`、`ApiError`）。
4. **`src/components/`**  
   `layout/` 管理台壳层；`feature-module-stub.tsx` 为各业务页统一占位；`ui/` 仅放 shadcn 生成组件。
5. **`src/pages/`**  
   仅保留与「领域无关」的全局页（如 404）。

路由侧使用 **`React.lazy` + `Suspense`**：在 `AppShell` 内对 `<Outlet />` 包裹 `Suspense`，fallback 使用 `PageLoading`。

## 4. ESLint 与热更新相关说明

- **`src/components/ui/**`**：shadcn 的 `button` 等会导出 `buttonVariants` 等非组件符号，与 `react-refresh/only-export-components` 冲突，故对该目录关闭该规则。
- **`src/app/router.tsx`**：存在多处 `lazy(...)` 工厂写法，同样关闭该规则。
- **`src/lib/auth/auth-context.tsx`**：同时导出 `AuthProvider` 与 `useAuth`，关闭 `react-refresh/only-export-components`。

## 6. 本地对接 Nest 建议

1. 在 `frontend` 下创建 `.env`（勿提交密钥；`.gitignore` 已默认忽略常见 env 文件时请自行核对）：

   ```env
   VITE_API_BASE_URL=http://localhost:3000/api
   ```

   端口按实际 Nest 监听修改。

2. 开发阶段可在 Vite 配置 **`server.proxy`**，将 `/api` 代理到后端，避免浏览器跨域；当前仓库若未配置代理，则需后端开启 CORS 或自行补充 `vite.config.ts`。

3. 业务请求统一经 **`apiRequest`**，各领域的具体路径写在对应 **`features/<领域>/api.ts`**，与技术设计文档中的模块划分保持一致。

4. **鉴权（方案 B）**：`POST /auth/login` 成功后把 `access_token` 写入 **`sessionStorage`**（见 `lib/auth/access-token.ts`）；`apiRequest` 自动附加 **`Authorization: Bearer`**。收到 **401** 时会清空令牌并派发 `harvestapp:auth-expired`，`AuthProvider` 同步状态后由 **`RequireAuth`** 重定向到 **`/login`**。侧栏提供「退出登录」。
