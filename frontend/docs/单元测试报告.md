# HarvestApp · 前端单元测试报告

**生成日期**：2026-05-05  
**适用范围**：`frontend/` 目录；单测由 **Vitest** 执行，配置位于根目录 **`vite.config.ts`** 的 `test` 段（`include`：`src/**/*.test.ts`、`src/**/*.test.tsx`）。**Playwright e2e**（`e2e/*.spec.ts`）与 Vitest 分离，不在本报告的用例统计内。

## 文档概述

本报告汇总前端 **单元测试与覆盖率** 的现状，并给出 **配置说明、目录结构、Mock 思路与常见问题**，便于代码评审、发版说明或 CI 对照。

**测试分层（建议）**：

| 层级 | 工具 | 典型覆盖对象 |
|------|------|----------------|
| 单元测 | Vitest + Testing Library | `lib/` 工具、`features/*/api.ts` 请求封装、纯函数（如 `groupAssignableRowsToTrackProjects`） |
| 组件测 | Vitest + jsdom + `@testing-library/react` | 关键页面加载、表单交互（当前仅 tasks 页有示例） |
| 端到端 | Playwright | 登录、导航、关键业务流程（见 `e2e/` 与 E2E 文档） |

**与 `单元测试说明.md` 的关系**：说明文档偏 **日常开发速查**；本报告偏 **某次里程碑下的执行数据、配置逐项解释与问题排查**，两者可交叉引用。

建议在较大范围补充用例或发版前执行 `npm run test:run` 与 `npm run test:coverage`，并更新本文 **「测试执行摘要」「覆盖率摘要」「详细用例清单」** 中的数字与日期。

## 2. 配置说明（Vitest / Vite）

配置集中在 **`frontend/vite.config.ts`**，使用 `defineConfig` 的 `test` 字段（由 `vitest/config` 提供类型）。

| 选项 | 当前行为 | 说明 |
|------|----------|------|
| `test.environment` | `jsdom` | 组件测试需要 `document`、`window` 等 |
| `test.setupFiles` | 绝对路径指向 **`vitest.setup.ts`** | 全局注册 `@testing-library/jest-dom/vitest` |
| **`test.include`** | **`src/**/*.test.ts`**、**`src/**/*.test.tsx`** | 仅 `src` 下匹配文件；`e2e/` 不参与 Vitest |
| `test.coverage.provider` | `v8` | 与 `@vitest/coverage-v8` 一致 |
| `test.coverage.reporter` | `text`、`html`、`json` | 终端摘要 + 浏览器 HTML + 机器可读 JSON |
| **`test.coverage.reportsDirectory`** | **`frontend/coverage/`**（`path.resolve(__dirname, 'coverage')`） | HTML 入口一般为 **`coverage/index.html`** |
| `resolve.alias` | **`@` → `./src`** | 与业务代码一致；测试中 `vi.mock('@/lib/...')` 依赖该别名 |

**环境变量与 `import.meta.env`**：业务侧通过 `src/lib/api/public-env.ts` 读取 `VITE_API_BASE_URL`。在 Vitest 中可使用 **`vi.stubEnv('VITE_API_BASE_URL', ...)`** 配合 **`vi.resetModules()`** 与动态 `import()` 做隔离测试（见 `public-env.test.ts`）。

**npm 脚本中的覆盖率收窄**：`test:coverage:tasks` 使用 `--coverage.include=src/features/tasks/**` 并限定路径为 `src/features/tasks`，适合按模块迭代时不被全仓未测文件拉低观感。

| 命令 | 作用 |
|------|------|
| `npm test` | Vitest **watch** 模式 |

## 5. Mock 策略（摘要）

- **`features/*/api.ts`**：对 `@/lib/api/http` 使用 **`vi.mock`**，将 **`apiRequest`** 设为 **`vi.fn()`**，用 **`mockResolvedValue`** 断言路径、`method`、`body` 及请求头；**不发起真实 fetch**。
- **`lib/api/http.ts`**：对 **`fetch`** 使用 **`vi.stubGlobal('fetch', vi.fn())`**，并 **`vi.mock`** `getPublicApiBaseUrl`、`getAccessToken`、`clearAccessToken`、`notifyAuthExpired`，验证 JSON 解析、401 清理 token、错误体 `message` 等分支。
- **bcrypt / 外部 SDK**：前端不在此仓库做单测；认证相关仅测 **请求拼装**。
- **带 TanStack Query 的页面**：使用 **`QueryClientProvider`**，可调低 **`queries.retry`**；可对具体 `api` 模块 **`vi.mock(..., importOriginal)`** 局部替换。

## 6. 常见问题排查

| 现象 | 可能原因 | 处理建议 |
|------|----------|----------|
| **`No test files found`** | 文件不在 `src/` 或扩展名不是 `.test.ts(x)` | 对照 `vite.config.ts` 的 `test.include` |
| **`Failed to resolve import "@/..."`** | Vitest 未继承 Vite 别名 | 确认 `vite.config.ts` 中 `resolve.alias` 存在且路径正确 |
| **`import.meta.env` 在测试中不符合预期** | 模块在首次 import 时已缓存 | 使用 **`vi.stubEnv`** + **`vi.resetModules()`** + 动态 **`import()`**（见 `public-env.test.ts`） |
| **组件测试找不到 role / 文本** | 语义化不足或异步未 `await` | 使用 `findBy*`、`waitFor`，或检查是否被 `aria` 隐藏 |
| **覆盖率全仓仍然很低** | 大量 `pages/*.tsx`、布局、路由未写单测 | 正常；可先 **`test:coverage:tasks`** 或按 feature 收窄 `coverage.include` |
| **watch 模式在 CI 挂起** | 误用 `npm test` | CI 使用 **`npm run test:run`** 或 **`vitest run`** |

## 8. 代码覆盖率摘要（`npm run test:coverage`）

全量收集下，**页面与路由文件**占源码体量很大，**语句/行覆盖率**会长期偏低；本次通过补充 **`lib/api/http`、`lib` 工具与多模块 API 封装** 等，全量语句覆盖率由约 **3.1%** 提升至约 **4.4%**；**函数覆盖率**提升至约 **30.6%**（v8 报表为全路径汇总，含未测文件）。

| 维度（All files，报告截取） | 比例（约） |
|-----------------------------|------------|
| Statements | **4.41%** |
| Branches | **56.42%** |
| Functions | **30.59%** |
| Lines | **4.41%** |

**部分文件参考（便于对照 HTML）**：

- `src/lib/utils.ts`、`src/lib/currencies.ts`、`src/lib/api/public-env.ts`、`src/features/auth/api.ts`：**语句 100%**
- `src/lib/api/http.ts`：约 **90%** 语句（余下为 JSON 解析失败、非对象 `message` 等分支）
- `src/features/time/api.ts`：语句比例显著高于 time 模块内未测的 `time-format.ts` / `time-range.ts`
- `src/features/tasks/api.ts`：`downloadTasksExport` 等仍依赖真实 `fetch`/`document`，当前未覆盖
