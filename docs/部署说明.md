# HarvestApp 生产环境部署说明

本文说明如何将 **HarvestApp**（前端 + Nest 后端 + PostgreSQL）通过 **Docker Compose** 部署到一台新的 Linux 服务器。部署形态与项目根目录的 `docker-compose.yml` 一致：浏览器访问 **Nginx（前端）**，接口走同源 **`/api`** 由 Nginx 反代到后端。

---

## 1. 服务器要求

| 项目 | 建议 |
|------|------|
| 系统 | Linux x86_64（如 Ubuntu 22.04、Debian 12） |
| 内存 | ≥ 2GB（同时跑 DB + 后端 + 前端） |
| 磁盘 | ≥ 20GB 可用空间（含镜像与数据卷） |
| 软件 | [Docker Engine](https://docs.docker.com/engine/install/) 与 [Docker Compose V2](https://docs.docker.com/compose/install/)（`docker compose` 子命令） |
| 网络 | 可出网拉取镜像；对外开放 **80/443**（或你映射的端口）用于访问应用 |

> **ARM 服务器**（如部分云主机）：需自行确认镜像能构建通过；若遇原生模块问题，以实际构建日志为准。

---

## 2. 部署前准备

### 2.1 将代码放到服务器

任选其一：

- `git clone` 仓库到服务器目录，或  
- 本机打包上传（**不要**把含真实密码的 `backend/.env` 提交进 Git，生产环境用环境变量或服务器侧 `.env`）

部署使用的目录结构应包含（至少）：

- 根目录：`docker-compose.yml`
- 子目录：`backend/`（含 `Dockerfile`、`prisma/`）、`frontend/`（含 `Dockerfile`、`nginx.conf`）

### 2.2 环境变量（必改项）

在 **生产环境** 必须设置强随机的 **JWT 密钥**，不要使用 compose 里的默认值。

**方式 A：在部署目录创建 `.env`（与 `docker-compose.yml` 同级）**

```env
# 与 docker-compose 中 ${JWT_SECRET} 对应
JWT_SECRET=请替换为至少 32 字节的随机串
JWT_EXPIRES_IN=7d
```

**方式 B：在 shell 中导出后再启动**

```bash
export JWT_SECRET='你的长随机串'
export JWT_EXPIRES_IN=7d
docker compose up -d --build
```

**数据库密码**：当前 `docker-compose.yml` 中 PostgreSQL 为默认 `postgres/postgres`，**公网部署务必修改**（需同时改 `postgres` 服务环境变量与 `backend` 的 `DATABASE_URL`，保持账号密码一致）。

## 3. 首次部署步骤

在 **项目根目录**（含 `docker-compose.yml`）执行：

```bash
cd /path/to/HarvestApp

# 生产建议：已设置 JWT_SECRET 等环境变量
docker compose up -d --build
```

- 首次会构建 `backend`、`frontend` 镜像，并启动 `postgres`、`backend`、`frontend`。
- 后端容器启动时会执行 **`npx prisma migrate deploy`**，将数据库结构迁移到最新（见 `backend/Dockerfile` 的 `CMD`）。

### 3.1 开放端口

默认将 **本机 8080** 映射到前端 Nginx 的 80 端口：

- 浏览器访问：`http://<服务器IP>:8080`  
- API 文档：`http://<服务器IP>:8080/api/docs`  

若需 **80 端口** 直接访问，可将 `docker-compose.yml` 中前端的 `ports` 改为 `"80:80"`（需 root 或具备绑定特权端口权限）。

### 3.2 种子数据（可选）

创建演示账号（邮箱 / 密码见 `backend/prisma/seed.js`，当前为 `demo@harvest.app` / `demo123`）：

```bash
docker compose exec backend npx prisma db seed
```

仅内网/测试使用；**生产环境**应关闭开放注册、使用强密码策略，并视情况**不要**执行 seed 或改 seed 内容。

## 4. 版本更新（重新发布）

代码更新后，在服务器项目根目录执行：

```bash
git pull   # 若用 Git
docker compose up -d --build
```

需要**无缓存**彻底重建时：

```bash
docker compose build --no-cache
docker compose up -d
```

只改后端或只改前端时，可只构建对应服务，例如：

```bash
docker compose build backend
docker compose up -d backend
```

## 5. 数据持久化与备份

| 数据 | Docker 卷名（默认） | 说明 |
|------|---------------------|------|
| PostgreSQL 数据 | `harvestapp_pgdata` | 业务与迁移状态均在此，**务必备份** |
| 上传文件（如收据） | `backend_uploads` | 映射到容器内 `/app/uploads` |

**查看卷：**

```bash
docker volume ls | grep harvestapp
```

**备份数据库（示例：从 postgres 容器 dump）**

```bash
docker compose exec -T postgres pg_dump -U postgres harvestapp > harvestapp_$(date +%Y%m%d).sql
```

**恢复**需在停写或维护窗口按 PostgreSQL 标准流程导入，此处不展开。

## 6. HTTPS 与域名（建议）

当前 Compose 为 **HTTP**。生产建议：

- 在宿主机或独立机器上部署 **Nginx / Caddy / Traefik**，对域名做 **TLS 终止**，反代到 `127.0.0.1:8080`（或你映射的端口）。  
- 或把证书挂到最外层反向代理，**不要**把私钥写进 Git。

前端构建时 `VITE_API_BASE_URL=/api`，走**同源**相对路径，一般**无需**为 API 单独配跨域公网地址；若你改为全量 URL，需同步改 `frontend` 构建参数与 CORS（`backend` 的 `main.ts` 中 CORS 需允许你的线上域名）。

## 7. 常用运维命令

```bash
# 查看运行状态
docker compose ps

# 查看日志（全部）
docker compose logs -f

# 只看后端
docker compose logs -f backend

# 停止
docker compose down

# 停止并删除数据卷（危险：会删库）
# docker compose down -v
```

## 8. 故障排查

| 现象 | 可检查项 |
|------|----------|
| 502 / 无法访问 | `docker compose ps` 是否均为 `running`；`docker compose logs backend` 是否有 `migrate` 或数据库连接错误 |
| 登录失败、无用户 | 是否未执行 `seed`；或数据库为新库，需注册或 seed |
| 构建失败 | 宿主机 Docker 与磁盘空间；`docker compose build --no-cache` 看完整错误 |
| 时区/时间不对 | 容器时区、PostgreSQL 与应用 `JWT_EXPIRES_IN` 预期 |

## 9. 安全清单（简）

- [ ] 修改 `JWT_SECRET` 与**数据库**默认密码。  
- [ ] 公网只暴露**必要端口**；用防火墙/安全组限制来源。  
- [ ] 生产 `NODE_ENV=production`（compose 中已设）。  
- [ ] 定期备份 `harvestapp_pgdata` 与重要上传数据。  
- [ ] 审查 `AUTH_ALLOW_REGISTRATION` 等环境变量（若后续通过 `env` 注入后端）。

---

## 10. 不通过 Docker 的部署（简述）

若目标环境**不能**用 Docker，需要：自行安装 **Node 22+**、**PostgreSQL 16+**，在 `backend` 执行 `npm ci`、`npx prisma generate`、`npm run build`、设好 `DATABASE_URL` 后 `npx prisma migrate deploy`，以进程管理器运行 `node dist/main.js`；`frontend` 执行 `npm ci`、`npm run build` 后由 **Nginx** 托管 `dist/`，并配置把 `/api` 代理到后端。此方式与仓库内 `Dockerfile` 行为应对齐，维护成本通常高于 Compose 一键部署。
