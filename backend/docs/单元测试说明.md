# HarvestApp · 后端单元测试说明

本文说明 `backend/` 中单测的**技术选型、目录约定、常用命令与 Mock 策略**，与 `Nest后端架构说明.md` 中的模块划分一致。

---

## 1. 技术栈

| 组件 | 说明 |
|------|------|
| Jest | 测试运行器与断言（见根目录 `package.json` 的 `jest` 字段） |
| ts-jest | 编译 TypeScript 测试文件 |
| `@nestjs/testing` | 构造 `TestingModule`、覆盖 Provider，便于测 Controller / Guard / Pipe |
| Supertest | 已在 `devDependencies` 中；若编写「启动 HTTP 应用」的集成测试时可配合使用 |

配置要点：

- **`rootDir`**：`src`，测试文件需放在 `src/` 下才会被默认 Jest 配置拾取。
- **`testRegex`**：`.*\.spec\.ts$`，即单测文件命名 **`*.spec.ts`**。

---

## 2. npm 脚本

| 命令 | 作用 |
|------|------|
| `npm test` | 运行全部单测 |
| `npm run test:watch` | 监听模式 |
| `npm run test:cov` | 带覆盖率 |
| `npm run test:tasks` | 只跑 `src` 下路径含 `modules/tasks` 的用例（Jest 30+ 使用 `--testPathPatterns`） |
| `npm run test:e2e` | 端到端测试，使用 `test/jest-e2e.json`（**不是**普通单元测默认配置） |
| `npm run test:debug` | 断点调试 Jest，见 Jest 文档 |

> **注意**：Jest 30 起，筛选文件请使用 CLI 参数 **`--testPathPatterns`**，旧版 `--testPathPattern` 已移除。

---

## 3. 目录与文件约定

| 类型 | 建议位置 | 说明 |
|------|----------|------|
| 领域单测 | 与源文件同目录，如 `src/modules/tasks/tasks.service.spec.ts` | 最常用；一眼能看出测的是哪个类 |
| 可复用 Test helper | `src/**/__tests__/` 或 `test/helpers/`（按团队习惯二选一） | 避免与 `*.spec.ts` 重名即可 |

**Tasks 模块示例**：`src/modules/tasks/tasks.service.spec.ts` 针对 `TasksService` 的纯逻辑与 Prisma 调用参数断言。

## 4. Mock 策略

### 4.1 Prisma / `PrismaService`（推荐）

- **单元测目标**：`Service` 内业务规则、错误分支、拼出来的 `where` / `data` 等。
- **做法**：对 `PrismaService` 做浅层 mock（`task.findMany`、`task.create` 等 **jest.fn()**），**不连真实数据库**。
- **何时用真实库**：可另建「集成测」或 e2e，在 CI 中配测试库；日常开发以 mock 为主、速度快、无环境依赖。

### 4.2 其它依赖

- **`OrganizationContextService`、其它模块的 Service**：在 `Test.createTestingModule` 里用 `useValue` / `useFactory` 提供假实现或 `jest.fn()`。
- **Controller**：可只测路由是否调用了 Service 正确入参，或起 `TestingModule` + `INestApplication` 做更贴近 HTTP 的测试（与 e2e 的界限按团队定义）。

### 4.3 与 e2e 的分工

- **单测 `*.spec.ts`（`rootDir: src`）**：无全局前缀下的「类 / 小模块」级验证。
- **`test/e2e/*.e2e-spec.ts` + `jest-e2e.json`**：全应用启动、真实路由、（可选）真实或测试数据库。

---

## 5. 与 Nest 官方模式对齐的提示

- 对 Service：`new Service(mockPrisma as PrismaService)` 或通过 `Test.createTestingModule` 注入 mock。
- 对需要装饰器元数据的类：优先用 `@nestjs/testing` 的 `TestingModule`，避免手写 `Reflect`。
- 覆盖率与 `collectCoverageFrom` 见根 `package.json` 的 Jest 配置，输出目录默认为项目根的 `coverage/`（相对 `rootDir: src` 的 `../coverage`）。
