# HarvestApp · 后端单元测试说明

本文说明 `backend/` 中单测的**技术选型、目录约定、常用命令与 Mock 策略**，与 `Nest后端架构说明.md` 中的模块划分一致。

---

## 1. 技术栈

| 组件 | 说明 |
|------|------|
| Jest | 测试运行器与断言（见根目录 `package.json` 的 `jest` 字段） |
| ts-jest | 编译 TypeScript 测试文件 |
| `@nestjs/testing` | 构造 `TestingModule`、覆盖 Provider，便于测 Controller / Guard / Pipe |
| Supertest | 已在 `devDependencies` 中；若编写「启动 HTTP 应用」的集成测试时可配合使用 |

配置要点：

- **`rootDir`**：`src`，测试文件需放在 `src/` 下才会被默认 Jest 配置拾取。
- **`testRegex`**：`.*\.spec\.ts$`，即单测文件命名 **`*.spec.ts`**。

---

## 2. npm 脚本

| 命令 | 作用 |
|------|------|
| `npm test` | 运行全部单测 |
| `npm run test:watch` | 监听模式 |
| `npm run test:cov` | 全量单测 + 覆盖率（`collectCoverageFrom` 见 `package.json` 的 Jest 配置，覆盖**整个** `src`） |
| `npm run test:cov:tasks` | 只跑 **tasks 模块** 用例 + 覆盖率；报告**仅统计** `src/modules/tasks/**/*.ts`（适合按模块迭代，避免未测模块拉低全量观感） |
| `npm run test:tasks` | 只跑 `src` 下路径含 `modules/tasks` 的用例，**不**收集覆盖率（Jest 30+ 使用 `--testPathPatterns`） |
| `npm run test:e2e` | 端到端测试，使用 `test/jest-e2e.json`（**不是**普通单元测默认配置） |
| `npm run test:debug` | 断点调试 Jest，见 Jest 文档 |

> **注意**：Jest 30 起，筛选文件请使用 CLI 参数 **`--testPathPatterns`**，旧版 `--testPathPattern` 已移除。

---

## 3. 目录与文件约定

| 类型 | 建议位置 | 说明 |
|------|----------|------|
| 领域单测 | 与源文件同目录，如 `src/modules/tasks/tasks.service.spec.ts` | 最常用；一眼能看出测的是哪个类 |
| 可复用 Test helper | `src/**/__tests__/` 或 `test/helpers/`（按团队习惯二选一） | 避免与 `*.spec.ts` 重名即可 |

**Tasks 模块示例**：`src/modules/tasks/tasks.service.spec.ts` 针对 `TasksService` 的纯逻辑与 Prisma 调用参数断言。

---

## 4. Mock 策略

### 4.1 Prisma / `PrismaService`（推荐）

- **单元测目标**：`Service` 内业务规则、错误分支、拼出来的 `where` / `data` 等。
- **做法**：对 `PrismaService` 做浅层 mock（`task.findMany`、`task.create` 等 **jest.fn()**），**不连真实数据库**。
- **何时用真实库**：可另建「集成测」或 e2e，在 CI 中配测试库；日常开发以 mock 为主、速度快、无环境依赖。

### 4.2 其它依赖

- **`OrganizationContextService`、其它模块的 Service**：在 `Test.createTestingModule` 里用 `useValue` / `useFactory` 提供假实现或 `jest.fn()`。
- **Controller**：可只测路由是否调用了 Service 正确入参，或起 `TestingModule` + `INestApplication` 做更贴近 HTTP 的测试（与 e2e 的界限按团队定义）。

### 4.3 与 e2e 的分工

- **单测 `*.spec.ts`（`rootDir: src`）**：无全局前缀下的「类 / 小模块」级验证。
- **`test/e2e/*.e2e-spec.ts` + `jest-e2e.json`**：全应用启动、真实路由、（可选）真实或测试数据库。

---

## 5. 与 Nest 官方模式对齐的提示

- 对 Service：`new Service(mockPrisma as PrismaService)` 或通过 `Test.createTestingModule` 注入 mock。
- 对需要装饰器元数据的类：优先用 `@nestjs/testing` 的 `TestingModule`，避免手写 `Reflect`。
- 全量 `test:cov` 的 `collectCoverageFrom` 与 `coverageDirectory` 见根 `package.json` 的 Jest 配置，输出目录为 **backend 根** 下 `coverage/`（Jest 中 `coverageDirectory` 相对 `rootDir: src` 为 `../coverage`）。**仅 tasks 带覆盖率**时脚本 `test:cov:tasks` 会在命令行**覆盖** `collectCoverageFrom`，与 `test:tasks` 的 `--testPathPatterns=modules/tasks` 搭配使用。

## 6. 可视化测试报告

### 6.1 代码覆盖率（推荐，**无需额外插件**）

| 说明 | 内容 |
|------|------|
| 全量 | `npm run test:cov`（即 `jest --coverage`） |
| 仅 tasks 模块 | `npm run test:cov:tasks`：等价于 `jest --coverage --testPathPatterns=modules/tasks --collectCoverageFrom=modules/tasks/**/*.ts`（路径均相对 Jest 的 `rootDir: src`） |
| 报告位置 | `backend/coverage/lcov-report/index.html`（由 **istanbul** 生成；全量与 tasks 专项都写入同一 `coverage` 目录，**后一次运行会覆盖**前一次结果） |
| 打开方式 | 用浏览器直接打开该 HTML 文件，可查看**按文件/行/分支**的覆盖情况 |

**全量与 tasks 专项的区别**：

- **`test:cov`**：跑全部 `*.spec.ts`，覆盖率按配置收集**整个** `src` 下匹配 `collectCoverageFrom` 的源码。
- **`test:cov:tasks`**：只执行路径匹配 `modules/tasks` 的测试；`--collectCoverageFrom` 收窄到 `modules/tasks/**/*.ts`，报表中**只出现** tasks 模块源文件，与「先集中补 tasks 单测」的工作流一致。

仅跑 tasks 用例、**不要**覆盖率时，仍用 **`npm run test:tasks`**（与前端「只跑用例」对应）。

若希望「跑完测试后自动打开报告」，可在本机用脚本包（如 `open-cli`）或系统命令在 **post-** 步骤里执行 `open` / `start`，跨平台需自行区分；**CI** 中通常只上传 `coverage` 目录为构建产物，由平台展示。

**其它模块**：仿照 `test:cov:tasks` 增加脚本，把 `--testPathPatterns` 与 `--collectCoverageFrom` 换成对应子路径即可（`collectCoverageFrom` 须与 `rootDir: src` 下目录一致，如 `modules/其它模块/**/*.ts`）。

### 6.2 用例通过/失败列表的 HTML（可选第三方）

Jest 默认在终端输出结果，**不**自带「整份测试套件的独立 HTML 汇总页」。若需要：

| 方向 | 说明 |
|------|------|
| **jest-html-reporter** | 安装后配置 `reporters`，可生成单页 HTML，含每个用例状态，适合「给人看」的归档 |
| **jest-junit** | 输出 JUnit XML，**偏 CI**（Jenkins、GitHub Actions 等展示），不是浏览器里那种图文报告 |
| **Allure** 等 | 与 `allure-jest` 等配合，功能强但依赖与配置更重 |

以上均需在 `package.json` 的 Jest 配置中增加 `reporters` 或 `testResultsProcessor`（以各包文档为准）。**本仓库当前未预装**上述 HTML 用例报告依赖，按项目需要再引入即可。
