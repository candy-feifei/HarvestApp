**生成日期**：2026-05-05

## 文档概述

本报告用于说明 **HarvestApp 后端** 单元测试的现状、配置方式、目录约定与一次完整执行结果摘要，便于评审、发版归档或对接 CI。单元测试聚焦 **Service 业务规则、工具函数与异常处理** 等可在 Node 中快速运行的逻辑；**HTTP 路由、Guard、真实数据库** 等更适合放在 e2e（`npm run test:e2e`）或后续集成测试中。

**与 `单元测试说明.md` 的关系**：

| 文档 | 侧重 |
|------|------|
| **`docs/单元测试说明.md`** | 面向日常开发的约定、命令速查、Mock 思路、覆盖率 HTML 打开方式 |
| **`docs/单元测试报告.md`（本文）** | 某次里程碑下的执行与覆盖率快照、配置逐项说明、目录结构、问题排查与用例清单 |

建议在每次较大范围补充用例或发版前：本地执行 `npm test` 与 `npm run test:cov`，并视需要更新本文 **「测试执行摘要」「覆盖率摘要」「详细用例清单」** 中的数字与日期。

## 2. Jest 配置说明

配置位于 **`backend/package.json`** 的顶层 **`"jest"`** 字段（未单独使用 `jest.config.js`），与 Nest CLI 默认脚手架风格一致。

| 字段 | 值 | 含义 |
|------|-----|------|
| `moduleFileExtensions` | `js`, `json`, `ts` | 参与解析的扩展名 |
| **`rootDir`** | **`src`** | 工作根为 `backend/src`；`testRegex`、`collectCoverageFrom` 等路径均相对此目录 |
| **`testRegex`** | **`.*\\.spec\\.ts$`** | 仅匹配名为 `*.spec.ts` 的文件；**`backend/test/` 下的 e2e 不会被本配置拾取** |
| `transform` | `ts-jest` | `.ts` / `.js` 经 ts-jest 转换 |
| **`collectCoverageFrom`** | **`**/*.(t|j)s`** | 覆盖率统计覆盖 `src` 下所有 `.ts`/`.js`（含未被任何测试命中的文件，故全量比例会「如实」反映缺口） |
| **`coverageDirectory`** | **`../coverage`** | 相对 `rootDir`（即 `src`）向上一级，报告输出到 **`backend/coverage/`** |
| `testEnvironment` | `node` | 默认 Node 环境，无浏览器 DOM |

**路径心智模型**：

- 在仓库根执行：`cd backend && npm test`。
- HTML 覆盖率报告：`backend/coverage/lcov-report/index.html`（执行 `npm run test:cov` 后生成）。

**按模块收窄覆盖率**（与 `单元测试说明.md` 一致）：脚本 `test:cov:tasks` 在命令行追加 `--collectCoverageFrom=modules/tasks/**/*.ts` 与 `--testPathPatterns=modules/tasks`，仅统计 tasks 模块源码；其它模块可仿照增加 npm script。

## 3. npm 脚本

| 命令 | 作用 |
|------|------|
| `npm test` | 运行 `src` 下全部 `*.spec.ts` |
| `npm run test:watch` | 监听模式 |
| `npm run test:cov` | 全量单测 + 覆盖率（写入 `backend/coverage/`） |
| `npm run test:cov:tasks` | 仅 tasks 模块用例 + 仅统计 `src/modules/tasks/**/*.ts` |
| `npm run test:tasks` | 仅路径含 `modules/tasks` 的用例（无覆盖率） |
| `npm run test:e2e` | 使用 `test/jest-e2e.json` 的端到端测试 |
| `npm run test:debug` | `node --inspect-brk` 挂接 Jest，便于断点调试 |

## 5. Mock 策略（摘要）

- **Prisma**：对本次用例会触达的 delegate（如 `user.findUnique`、`task.findMany`）使用 `jest.fn()`，**不连接真实数据库**；断言关注 `where` / `data` 形状与分支行为。
- **bcrypt**（`auth.service.spec.ts`）：使用 `jest.mock('bcrypt')` 固定 `hash` / `compare`，避免加密耗时与实现细节干扰业务断言。
- **Config / Jwt / Mail**：以最小假对象或 `jest.fn()` 注入，只覆盖与分支相关的返回值。
- **与 e2e 分工**：单测保证模块内逻辑；全链路、真实路由与可选真实库放在 **`npm run test:e2e`**。

## 6. 常见问题排查

| 现象 | 可能原因 | 处理建议 |
|------|----------|----------|
| **`No tests found`** 或改了测试却不执行 | 文件不在 `src/` 下，或不匹配 `*.spec.ts` | 将单测移入 `src` 并检查文件名；e2e 须用 `test:e2e` |
| 报错 **`Unknown option: testPathPattern`** | Jest 30+ 已移除旧 CLI 参数 | 改用 **`--testPathPatterns`**（如 `npm run test:tasks` 已使用） |
| **`Cannot find module`** 或路径别名失败 | `rootDir` 为 `src` 时路径习惯与运行 cwd 不一致 | 在 `backend` 目录执行脚本；检查 `tsconfig` paths 是否与 Jest 一致（本仓库未额外配置 `moduleNameMapper`） |
| **`bcrypt` / 原生模块报错 | 在错误环境跑 Jest 或 mock 未生效 | 认证相关单测已 mock `bcrypt`；若扩写其它用例，保持 `jest.mock('bcrypt')` 在文件顶部 |
| 覆盖率**全量仍然偏低** | `collectCoverageFrom` 包含大量未写单测的 Controller / Module / DTO | 属预期；可短期用 **`test:cov:某模块`** 聚焦迭代，或继续按 Service / util 补用例 |
| **`coverage/` 与上次结果不一致** | 多次运行会覆盖同一输出目录 | 发版前保留一次 `test:cov` 的 HTML 或 lcov 产物即可 |

## 10. 未纳入单测或覆盖仍较少的部分

以下区域在工程中体量较大或偏「装配 / 路由」，当前**未以单测完整覆盖**，适合按优先级逐批补充或用 e2e 兜底：

- **Controller 层**（各模块 `*.controller.ts`）：可结合 `@nestjs/testing` + `supertest` 做轻量集成测，或依赖 e2e。
- **Guards / Strategies**（如 `jwt-auth.guard.ts`、`jwt.strategy.ts`）。
- **大型 Service**（如 `organization.service.ts`、`time-entries.service.ts`、`approvals.service.ts`、`reports.service.ts` 的主体报表逻辑）：建议按 **纯函数抽取 + mock Prisma** 分段推进，避免单文件过大难以维护。
- **settings、uploads、access** 等模块：与 `单元测试说明.md` 中描述一致，可按路由或关键 Service 补测。
