## 1. 目标与边界

### 1.1 我们用 AI 做什么

- **需求与拆解**：把业务需求拆成可实现的任务、接口与验收标准。
- **工程化实现**：按仓库既有结构新增/修改模块（Nest module、React feature、DTO、API 调用、Query 缓存）。
- **测试与文档**：生成单测/E2E 思路与用例骨架；维护可复用的 `prompts/` 模板与 `docs/` 文档。
- **代码评审与复盘**：生成 Review checklist、PR 描述草稿、风险清单与答辩 Q&A。

### 1.2 AI 不该替代的事情（必须人工拍板）

- **需求优先级与范围裁剪**：短周期下的 P0/P1 取舍。
- **权限与安全**：RBAC/数据隔离/越权风险，任何“代操作他人数据”必须严格审查。
- **数据模型与迁移策略**：表结构、索引、约束与回滚策略。
- **关键 UX 决策**：信息架构、关键流程是否符合目标用户。

## 2. 安全与脱敏（必须遵守）

### 2.1 禁止提交/粘贴的内容

- **密钥/Token/账号密码**：`.env`、JWT、数据库连接串、第三方 API key。
- **真实客户/个人隐私**：姓名、邮箱、电话、地址、合同、内网 URL、截图中可识别信息。
- **公司内部信息**：未公开业务、内部代号、服务器地址、日志原文（含敏感字段）。

### 2.2 允许提交的内容（推荐）

- **模板 + 占位符**：将真实信息替换为 `ACME_CORP`、`USER_A`、`TOKEN_XXX` 等。
- **已脱敏示例**：能展示 Prompt 结构与方法，但不含敏感数据。
- **PR/Commit 可对照的产出**：在 PR 描述中引用使用过的模板文件名与人工校验点。

### 2.3 在对话/工具里交给模型前（10 秒自检）

- 是否已去掉环境变量、Cookie、Authorization 头、连接串？  
- 堆栈/日志里是否带邮箱、用户 ID 以外的 PII？  
- 截图是否已打码或换示意图？

## 3. 推荐工作流（最省返工）

### 3.1 “先契约后实现”

1. 用 `prompts/prd.md` 产出：**P0/P1 功能清单 + 验收标准**。
2. 用 `prompts/nest-module.md` 明确：**API 路径、DTO 字段、错误码/异常文案、权限规则**（面向用户的错误信息须 **可读、不泄露实现细节**；团队若约定英文 UI/接口提示，在 PRD 中写清）。
3. 若需要，先跑通 **最小链路**（登录 → 列表 → 新增/编辑 → 审批/提交等）。
4. 最后再补 **UI 打磨 / 边界条件 / 测试**。

### 3.2 “小步快跑 + 可回滚”

- 每次改动尽量围绕一个明确目标：一个 API、一个页面、一个交互或一条测试链路。
- 任何影响范围大的改动，都要写清：  
  - **涉及文件**（路径列表）  
  - **风险点**（权限、缓存、数据迁移）  
  - **回滚方式**（撤销哪些提交/恢复哪些文件）

### 3.3 实现后最小验证（与 AI 协作时别省）

- 前端：在子项目下执行 `npm run build` 或 `npx tsc -b`，保证类型通过（具体脚本以 `frontend/package.json` 为准）。  
- 后端：在 `backend/` 下按项目脚本跑 build/test，避免合入后才发现类型或编译错误。  
- 有变更的接口：至少手测或在 PR 中说明用 curl/客户端如何验证。

## 4. 本项目结构约定（给 AI 的上下文要写清）

### 4.1 前端（`frontend/`）

- 领域按 `src/features/<domain>/` 组织；页面多在 `pages/`，接口在 `api.ts`。
- HTTP 统一封装在 `src/lib/api/http.ts`（`apiRequest` + `ApiError` + 401 处理）。
- 数据获取优先用 **TanStack Query**；Query key 必须包含影响数据的维度（如 week、view、scope、被查看用户等），避免 **缓存串数据**。

### 4.2 后端（`backend/`）

- NestJS 按模块组织：`src/modules/<domain>/...`。
- DTO 在模块内 `dto/`；通用能力在 `src/common/`（decorators/filters/utils/dto）。
- 任何“跨用户/代操作”能力必须：  
  - **显式参数**（query/header/body），不可隐式只依赖“看起来像当前用户”的侧信道  
  - **权限校验**（角色/能力）  
  - **组织成员校验**（同组织、状态合法等，以各模块实际规则为准）

### 4.3 在 Prompt 里建议同时给出的信息

- **具体路径**（如 `frontend/src/features/time/api.ts`）比只说“时间模块”更省轮次。  
- **已有类似实现**：指向可参考的文件/模块，降低风格漂移。  
- **不改动范围**（如“不要动迁移脚本/不要改 public API 路径”），减少无关重构。

## 5. 如何向 AI 提需求（Prompt 写法规范）

### 5.1 必填信息（建议复制粘贴）

- **目标**：一句话描述要达成的用户价值。  
- **范围**：影响哪些页面/API/模块（路径或模块名）。  
- **权限**：谁能看/谁能改/是否涉跨用户与组织隔离。  
- **验收标准**：至少 3 条，**可执行、可测**。  
- **约束**：不改哪些文件/须保持的兼容面（API、字段、行为）。

### 5.2 有 bug 时额外附上

- **现状**：用户或日志里**脱敏**后的表现（无敏感 token/真实客户名）。  
- **期望**：一句话 + 边界（空数据、无权限、并发等若相关）。  
- **复现**：环境（本地/哪个分支）与最短步骤。  

### 5.3 推荐模板（直接用仓库文件）

- 需求/验收：`prompts/prd.md`  
- Nest 模块与接口：`prompts/nest-module.md`  
- 前端 Feature：`prompts/frontend-feature.md`  
- 测试：`prompts/testing.md`  
- 复盘/答辩：`prompts/review.md`  

> 团队协作建议：PR 描述里写上 **使用的模板文件名 + 人工校验点**，便于复盘与比赛材料整理。

## 6. 人工校验清单（每次合入前至少过一遍）

### 6.1 通用

- [ ] 无敏感信息提交（`.env`、token、真实数据、截图敏感内容）  
- [ ] 路由/API/字段命名与现有代码风格一致  
- [ ] 错误信息对用户可读（**避免**直接暴露堆栈、内部表名/字段名/路径）  
- [ ] 关键逻辑有最小测试覆盖（单测或 E2E 或明确说明为何本次豁免）  
- [ ] 若产品/团队约定**英文（或其它固定语言）** UI/提示，新文案与现网一致、无未翻译残留  

### 6.2 权限与数据隔离（涉及团队/他人数据时必选）

- [ ] 非授权角色无法越权查看/修改他人或他组织数据  
- [ ] 任何“以某身份代操作/查看”的入口有明确权限与审计痕迹（按产品设计）  
- [ ] 前端 Query key 含**区分不同数据范围** 的因子，避免列表/详情串缓存  
- [ ] UI 对越权/只读/待审批等有清晰提示（如 timesheet 锁定、多入口冲突说明等）

## 7. 产出物与比赛材料如何对齐

- **代码**：PR/Commit 记录可追溯；避免无关的大杂烩提交。  
- **模板**：`prompts/` 中保留可复用模板与脱敏示例；迭代见 [`prompts/README.md`](./README.md) 的约定。  
- **说明文档**：`docs/` 或子项目 `docs/` 中补充架构/接口/联调说明，便于新成员与评委跟进。  
- **复盘**：用 `prompts/review.md` 生成比赛复盘要点与 Q&A，再人工补充证据（截图/录屏/PR 链接）。

## 8. 反模式与常见返工

- **只描述“做个 XXX”**、不给出模块路径与验收，导致多轮确认与风格不一致。  
- **把 .env/密钥/整段生产日志**贴进对话或提交到仓库。  
- **大拆大建**：同一 PR 里混需求变更、大重构、格式化全仓库，难 review、难回滚。  
- **只改前端或只改后端**：契约未对齐就实现，联调时才发现 DTO/状态码不一致。  
- **忽略 Query 维度**：多用户/多周/多视图下 key 过粗，出现界面显示错误数据。  
- **合入前未跑 typecheck/build**：在 CI 或他机才爆编译错误。

## 9. 文档与模板的维护

- 新类型任务若反复使用同类 Prompt，应**固化为** `prompts/*.md` 新文件或更新现有模板，并同步本说明中的链接与清单。  
- 重大架构或目录变更后，**至少**更新本文件第 4 节与 `prompts/README.md` 的「文件一览」表。  
- 本说明侧重「人与 AI 协作的规矩」；**具体技术细节**以代码与 `docs/` 为准，避免与实现漂移。

### 10.1 小功能 / 需求迭代（复制后替换）

```text
【目标】<一句话：用户/管理员能完成什么，价值是什么>

【范围】<前端路径或模块>；<后端模块或路由前缀>；<不在范围内请写清>

【权限】<谁可见/谁可操作>；<是否涉跨用户、代操作、多组织>；无则写「同现有 timesheet/项目 模块默认规则」

【验收（至少 3 条，可测）】
1. <操作 + 期望可见结果 / API 行为>
2. <边界：空列表、无权限、已锁定 等，若相关>
3. <错误提示或拦截方式，不泄露实现细节>

【约束】<不修改的文件或不得破坏的 API 行为>；<须跑的命令：如 frontend npm run build>

【参考】<可选：仓库内可照抄的类似文件路径>
```
