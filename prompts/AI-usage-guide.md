# HarvestApp 项目 AI 使用说明（团队协作版）

本文是本仓库的 **AI 使用说明**，用于比赛材料与团队协作统一口径。  
配套可复用 Prompt 模板位于 `prompts/`（见 `prompts/README.md`）。

---

## 1. 目标与边界

### 1.1 我们用 AI 做什么

- **需求与拆解**：把业务需求拆成可实现的任务、接口与验收标准。
- **工程化实现**：按仓库既有结构新增/修改模块（Nest module、React feature、DTO、API 调用、Query 缓存）。
- **测试与文档**：生成单测/E2E 思路与用例骨架；维护可复用的 `prompts/` 模板与 `docs/` 文档。
- **代码评审与复盘**：生成 Review checklist、PR 描述草稿、风险清单与答辩 Q&A。

### 1.2 AI 不该替代的事情（必须人工拍板）

- **需求优先级与范围裁剪**：两周周期下的 P0/P1 取舍。
- **权限与安全**：RBAC/数据隔离/越权风险，任何“代操作他人数据”必须严格审查。
- **数据模型与迁移策略**：表结构、索引、约束与回滚策略。
- **关键 UX 决策**：信息架构、关键流程是否符合目标用户。

---

## 2. 安全与脱敏（必须遵守）

### 2.1 禁止提交/粘贴的内容

- **密钥/Token/账号密码**：`.env`、JWT、数据库连接串、第三方 API key。
- **真实客户/个人隐私**：姓名、邮箱、电话、地址、合同、内网 URL、截图中可识别信息。
- **公司内部信息**：未公开业务、内部代号、服务器地址、日志原文（含敏感字段）。

### 2.2 允许提交的内容（推荐）

- **模板 + 占位符**：将真实信息替换为 `ACME_CORP`、`USER_A`、`TOKEN_XXX` 等。
- **已脱敏示例**：能展示 Prompt 结构与方法，但不含敏感数据。
- **PR/Commit 可对照的产出**：在 PR 描述中引用使用过的模板文件名与人工校验点。

---

## 3. 推荐工作流（最省返工）

### 3.1 “先契约后实现”

1. 用 `prompts/prd.md` 产出：**P0/P1 功能清单 + 验收标准**。
2. 用 `prompts/nest-module.md` 明确：**API 路径、DTO 字段、错误码/异常文案、权限规则**。
3. 若需要，先跑通 **最小链路**（登录 → 列表 → 新增/编辑 → 审批/提交）。
4. 最后再补 **UI polish / 边界条件 / 测试**。

### 3.2 “小步快跑 + 可回滚”

- 每次改动尽量围绕一个明确目标：一个 API、一个页面、一个交互或一条测试链路。
- 任何影响范围大的改动，都要写清：
  - **涉及文件**（路径列表）
  - **风险点**（权限、缓存、数据迁移）
  - **回滚方式**（撤销哪些提交/恢复哪些文件）

## 4. 本项目结构约定（给 AI 的上下文要写清）

### 4.1 前端（`frontend/`）

- 领域按 `src/features/<domain>/` 组织；页面通常在 `pages/`，接口在 `api.ts`。
- HTTP 统一封装在 `src/lib/api/http.ts`（`apiRequest` + `ApiError` + 401 处理）。
- 数据获取优先用 **TanStack Query**；Query key 必须包含影响数据的维度（如 week、view、subject user 等）。

### 4.2 后端（`backend/`）

- NestJS 按模块组织：`src/modules/<domain>/...`。
- DTO 在模块内 `dto/`；通用能力在 `src/common/`（decorators/filters/utils/dto）。
- 任何“跨用户/代操作”能力必须：
  - **显式参数**（query/header/body），不可隐式复用 JWT subject
  - **权限校验**（管理员/经理等）
  - **组织成员校验**（同组织、ACTIVE）

---

## 5. 如何向 AI 提需求（Prompt 写法规范）

### 5.1 必填信息（建议复制粘贴）

- **目标**：一句话描述要达成的用户价值
- **范围**：影响哪些页面/API/模块（路径或模块名）
- **权限**：谁能看/谁能改/是否允许代操作
- **验收标准**：至少 3 条，可测试
- **约束**：不改哪些文件/必须保持哪些接口兼容

### 5.2 推荐模板（直接用仓库文件）

- 需求/验收：`prompts/prd.md`
- Nest 模块与接口：`prompts/nest-module.md`
- 前端 Feature：`prompts/frontend-feature.md`
- 测试：`prompts/testing.md`
- 复盘/答辩：`prompts/review.md`

> 团队协作建议：PR 描述里写上“使用了哪个模板 + 人工校验点”，便于复盘与比赛材料整理。

---

## 6. 人工校验清单（每次合入前至少过一遍）

### 6.1 通用

- [ ] 无敏感信息提交（`.env`、token、真实数据、截图敏感内容）
- [ ] 路由/API/字段命名与现有代码风格一致
- [ ] 错误信息对用户可读（避免堆栈/内部字段泄露）
- [ ] 关键逻辑有最小测试覆盖（单测或 e2e 任一种）

### 6.2 权限与数据隔离（涉及团队/他人数据时必选）

- [ ] 非管理员/经理无法查看/修改他人数据
- [ ] 代操作必须验证目标用户属于当前组织且 ACTIVE
- [ ] 前端 Query key 含“被查看用户”，避免缓存串数据
- [ ] UI 有明确提示（例如切换到他人 timesheet 的 banner/warning）

---

## 7. 产出物与比赛材料如何对齐

- **代码**：PR/Commit 记录可追溯（不要把大改动混在一个提交里）。
- **模板**：`prompts/` 中保留可复用模板与脱敏示例。
- **说明文档**：`docs/`（或各子项目 `docs/`）里补上架构/接口/联调说明。
- **复盘**：用 `prompts/review.md` 生成比赛复盘要点与 Q&A，再人工校对与补充证据（截图/录屏/PR 链接）。
